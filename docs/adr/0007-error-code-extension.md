# 0007-error-code-extension

集成测试设计过程中发现handler返回None时缺乏明确的错误码定义。原有result码0-9未覆盖请求格式类错误（JSON解析失败、Base64解码失败）。为支持集成测试异常场景验证，扩展错误码体系，新增result=10（请求JSON解析失败）和result=11（Base64解码失败），形成清晰的错误分类：0-2验签通过、3-6验签失败类、7-9签名失败类、10-11请求格式类、≥12预留扩展。

## Considered Options

1. **合并到result=6**：将JSON解析失败、Base64解码失败合并到"格式错误(6)"，不区分错误层级
2. **按层级分类**：区分CMS格式错误(6)、JSON解析失败(10)、Base64解码失败(11)，分别对应数据层、协议层、编码层错误
3. **使用type区分**：通过vsock报文type字段区分错误类型，不扩展result码
4. **handler返回None触发框架层type=0x00/0x01**：依赖框架层处理，不返回业务result码

## Decision

选择**按层级分类（方案2）**，扩展result码为10/11。

理由：
- 错误层级清晰：CMS格式错误(6)为数据层、JSON解析失败(10)为协议层、Base64解码失败(11)为编码层
- 测试验证精细：集成测试可精确验证每种错误场景的result码
- handler有明确返回：无需依赖框架层猜测错误原因
- 错误码分类便于客户端诊断：按result码范围快速定位错误层级

## Consequences

- result码体系扩展为0-11+预留，CONTEXT.md和接口文档需同步更新
- handler实现需补充JSON解析失败和Base64解码失败的错误码映射
- 客户端可按result码分类快速定位错误层级：
  - 0-2: 验签通过的合法结果
  - 3-6: 验签失败类（证书链、CRL、签名、CMS格式）
  - 7-9: 签名相关失败类（证书、私钥、算法）
  - 10-11: 请求格式类错误（JSON、Base64）
  - ≥12: 其他错误（预留扩展）
- ADR-0001的错误码编码策略仍适用，新错误码延续分类逻辑
